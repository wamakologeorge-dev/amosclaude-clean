# Amosclaud Real Model Station

This deployment runs three cooperating services:

1. **Ollama** hosts the model checkpoint.
2. **Model Station** exposes fast `/health`, protected `/ready`, `/v1/models`, and
   OpenAI-compatible `/v1/chat/completions`.
3. **Station Runner** sends heartbeats, advertises `model.inference`, claims encrypted cloud requests, calls the local model, and returns the result.

## Start

```bash
cd deploy/model-station
cp .env.example .env
# Fill in the runner ID, runner token, and a long model token.
docker compose up -d --build
```

The bootstrap container downloads the configured model before the station starts.

## Required cloud configuration

The Amosclaud server must set:

```text
AMOSCLAUD_MASTER_KEY=<long production secret>
AMOSCLAUD_NETWORK_OWNER_USER_ID=<database user id that owns the runner>
```

The `.env` file on the station must contain the runner ID and token generated by Amosclaud. Never commit that file.

## Verify the station itself

```bash
curl -H "Authorization: Bearer $AMOSCLAUD_MODEL_TOKEN" http://localhost:8090/health
```

A real station returns `"status":"ready"`, the model name, backend, and measured latency from
`/ready`. `/health` remains a fast unauthenticated liveness response so Railway, Docker, and other
service managers do not kill a working process while a model loads.

The station accepts either `AMOSCLAUD_MODEL_TOKEN` or `AMOSCLAUD_API_KEY` through
`Authorization: Bearer ...`; it also accepts the Amosclaud API key through `X-API-Key`.
When Ollama or another configured upstream is missing, `AMOSCLAUD_FOLDER_MODEL_FALLBACK=true`
creates and trains the owned compact folder-native checkpoint from the packaged Amosclaud
bootstrap corpus. This fallback keeps functions available but is not represented as a frontier
model.

## Verify end-to-end

Open `/cloud/agent` and run the backend health check. The model planning service becomes ready only after:

- the runner heartbeat reaches Amosclaud,
- `model.inference` is advertised,
- the local station reports ready,
- the cloud owner user ID matches the runner owner.

This deployment does not pretend a model exists. Readiness depends on a successful inference probe.
